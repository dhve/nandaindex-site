Questions about the NANDA Index.

The short version of the architecture, the trust model, and how it fits with the protocols you already use. For the full detail, read Beyond DNS.

The basics

What is the NANDA Index? +

The NANDA Index is a lean, DNS-inspired index for the Internet of AI Agents. It maps an agent's identity to a tiny, cryptographically signed record called an AgentAddr, which in turn points to dynamic, verifiable AgentFacts.

Think of it as the layer that answers "who is this agent, where do I reach it, and can I trust what it claims?" without forcing the whole web to be rebuilt. NANDA stands for Networked AI Agents in a Decentralized Architecture.

Why can't DNS just handle AI agents? +

DNS, BGP and WHOIS were designed for millions of mostly static records that change on a minute-to-hour cycle. Autonomous agents are different: they may spawn, migrate and rotate endpoints every few seconds, and there can be billions of them.

DNS also only proves domain ownership. A TLS certificate says nothing about an agent's code, behaviour or whether its claimed capabilities are real. NANDA adds signed capability proofs, sub-second revocation, privacy-preserving lookups and an append-only audit trail that DNS was never meant to provide.

Is NANDA trying to replace DNS or the web? +

No. NANDA is deliberately additive. It introduces exactly one constraint-shifting primitive (a lean, signed record that resolves to verifiable facts) and leaves the layers that already scale, such as HTTPS, CDNs and existing registries, untouched.

The goal is a flexible substrate for agent discovery, routing and capability negotiation that sits alongside today's infrastructure rather than tearing it down.

Architecture

What are AgentFacts? +

AgentFacts are self-describing JSON-LD documents, each signed as a W3C Verifiable Credential. They carry the things that actually change: endpoint lists, capability descriptors, telemetry configuration and authentication methods.

Because facts live outside the index, an agent can update its capabilities and endpoints continuously without ever writing to the index. The index only stores a pointer (facts_url) and a signature.

What are the three layers of the architecture? +

1. Lean Index (anchor). A decentralized mapping from agent IDs to an AgentAddr of roughly 120 bytes, holding a DID, TTL, pointers and an Ed25519 signature.

2. AgentFacts (metadata). Signed JSON-LD documents the agent updates independently, split across a primary URL and a privacy-preserving private URL.

3. Dynamic Resolution (routing). Latency, privacy and resilience-aware endpoint selection using static endpoints (TTL 1 to 6 hours), an adaptive resolver for geo load-balancing and DDoS mitigation, and rotating endpoints (TTL 5 to 15 minutes).

What does the AgentAddr record actually contain? +

A minimal set of static fields: a decentralized identifier (id), a ttl, a facts_url, an optional private_facts_url, an adaptive_resolver_url and a signature. That is roughly all it takes to stay under 120 bytes per record.

Keeping the record this small is what lets the index avoid the write overhead that would otherwise make billion-agent scale impossible. Everything dynamic is delegated to AgentFacts.

What is the "quilt" model? +

The index is envisioned as a quilt stitched across many kinds of registries: NANDA-native agents, enterprise agent stores, government registries, Web3 marketplaces, IoT devices, MCP servers and NLWeb agents.

Some agents are registered directly and are fully visible. Others are administered by an external registry and are visible only to varying degrees, for example @company:shop versus a routed @company. NANDA does not need to govern every agent; it simply holds a verifiable pointer to the right discovery object.

Trust & privacy

What does "verify on claim, authenticate on update, attest on read" mean? +

It is the trust lifecycle of an entry. Verify on claim: when an entity claims an agent identity, ownership is verified. Authenticate on update: every change to the record or its facts must be cryptographically authenticated. Attest on read: when someone resolves the agent, they receive signed attestations they can independently check.

The effect is that trust is established once and re-checked continuously, rather than assumed because a lookup succeeded.

How does NANDA preserve privacy? +

Standard lookups leak who is asking for what, which enables passive tracking. NANDA uses dual-path resolution: a public PrimaryFactsURL and a PrivateFactsURL that can be hosted on third-party or decentralized storage (IPFS, Tor-style relays, a CDN or an enterprise bucket).

This supports least-disclosure, unlinkable queries and split-horizon governance, where different requesters can be given different answers, with tamper-evident audit logs. The trade-off is a small added latency, on the order of tens of milliseconds.

How fast is revocation and key rotation? +

Sub-second. Credentials reference a VC status list that can mark an agent's credentials revoked in under a second, and short-lived verifiable credentials (often under five minutes) limit the blast radius of any compromise.

This is a deliberate contrast with certificate ecosystems where revocation can take hours to propagate.

What stops an agent from lying about its capabilities? +

Capabilities are not self-asserted prose. They are schema-validated claims signed as W3C Verifiable Credentials, with issuers such as enterprises, consortia or federated certification authorities that can cross-sign each other within trust zones.

Each claim is anchored to an issuer ID with a revocation status, which directly targets capability spoofing, impersonation and supply-chain poisoning.

Ecosystem & usage

How does NANDA relate to MCP, A2A and NLWeb? +

They solve different problems. MCP (Model Context Protocol), A2A (Agent-to-Agent) and NLWeb describe how agents and tools communicate. NANDA is the layer underneath that handles discovery, identity and trust across all of them.

An MCP server, an A2A agent and an NLWeb agent can each appear as a patch in the NANDA quilt, which is what makes cross-protocol interoperability possible rather than leaving each ecosystem isolated.

Is the index centralized? +

No. The index is a decentralized, federated mapping, and AgentFacts are hosted by the agents and registries themselves on self-selected storage. Records are signed and cacheable, so resolution does not depend on a single authority being online.

Decentralized publishing is what lets the system absorb very high update rates without an index write bottleneck.

How do I register or resolve an agent? +

You can browse the live directory and resolve any agent identity to its discovery object at nandaindex.org. Registration is available to organizations and registries that want their agents indexed.

Under the hood, resolving an identity returns the AgentAddr, which you follow to the AgentFacts and then to a live endpoint via the dynamic resolution layer.

Who is building NANDA, and what is its status? +

NANDA is a research initiative led from MIT (Project NANDA), with collaborators across industry and academia including contributors from Cisco, CMU, TCS and others. It grew out of the MIT and Cisco Outshift collaboration on the agentic web.

The architecture is published as a working draft (v0.3, request for comments), so specifics are expected to evolve. Start with Beyond DNS and the enterprise perspective paper.
