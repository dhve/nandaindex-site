Basics

What is NandaIndex?+

A federated resolution layer. It is an AI Catalog-formatted database of index records, where each record maps a stable identity to the correct next discovery object. It introduces no new record format and does not host or execute agents.

Is NandaIndex a registry?+

No. A registry holds agent descriptions; NandaIndex holds pointers. An entry resolves an identity to wherever its description already lives, whether that is an AI Catalog, a DNS-AID record, a gateway, an AGNTCY ADS entry, or a hosted agent card.

Do I need NandaIndex if I already publish AI Catalog?+

Not for direct resolution. When you publish AI Catalog at a well-known endpoint and own the domain, any requester fetches it directly and NandaIndex is not in the critical path. An entry remains useful for fallback, federation visibility, migration, and anti-squatting.

What is a discovery object?+

The target an identity resolves to: an AI Catalog record or .well-known URL, a DNS-AID record, an AGNTCY ADS entry, an ANS record, an enterprise gateway, an A2A Agent Card or MCP descriptor, a hosted card, or a sector registry.

Entries and resolution

What does an entry contain?+

Four things: a stable identifier (URN or URI), the mediaType of the target, the target itself as a url or inline data, and metadata routing hints under org.projectnanda.*.

When does an entry use a URL versus inline data?+

A url is used when the target is fetchable directly, such as a hosted card or a well-known catalog. Inline data is used when resolution hands off to a system that owns the lookup itself, such as a DNS-AID agentDiscoveryName.

What is a resolution role?+

A metadata hint, org.projectnanda.resolutionRole, that describes how to treat the target. Examples include nested-ai-catalog, dns-aid-pointer, personal-agent-card, and gateway.

How does resolution proceed?+

NandaIndex maps the identity to a discovery object and returns its routing hints. The requester reads the discovery object, then the agent card for endpoint and auth, then calls the runtime. NandaIndex routes the first hop and then steps aside.

Identities and deployment

What identities are supported?+

Any stable URN or URI. Domain identities (urn:ai:domain:...), email identities (urn:ai:email:...), and system identities (urn:ai:system:...) are all entries of the same shape.

What is permissionless deployment?+

An identity does not require a unified infrastructure stack. The runtime, the agent card, and the brand domain can each live with a different provider. No single party controls all three, and no enterprise gateway or DNS certificate is required.

Can an SMB or individual register without a domain?+

Yes. An individual can use an email identity. An SMB can own a domain but delegate card hosting and runtime to third parties. An entry only needs a stable identity and a target.

What is host39.org?+

A third-party agent-card host. It lets SMBs and individuals publish agent-card information even when they do not own a DNS domain or cannot run catalog infrastructure on their own domain. The runtime and brand domain can live with other providers; host39.org serves the card, and a NandaIndex entry points to it.

Ecosystem

How does NandaIndex relate to AI Catalog?+

It uses the AI Catalog Catalog Entry as its index-record format and strengthens AI Catalog by making it the framework through which other discovery systems bridge.

How does it relate to AGNTCY ADS and ARD?+

ADS is a decentralized backend directory; ARD is a read-only finder API over conforming registries. An entry can point at an ADS entry, and ADS exposes an AI Catalog-compatible interface, so the same record is reachable by catalog lookup, ARD search, or NandaIndex resolution.

Does it replace DNS or DNS-AID?+

No. For domain-controlled, cacheable resolution, DNS-AID stays authoritative. NandaIndex makes the DNS-AID path reachable from the switchboard by pointing to it with inline routing data.

How do emerging registries (telecom, IoT, sovereign) join?+

Each publishes one entry pointing to its native mechanism. It does not adopt AI Catalog internally and does not change its existing system. The switchboard absorbs the new system without requiring conformance.
