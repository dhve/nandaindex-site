NandaIndex · Federated Resolution Architecture

A Global Switchboard for the Agentic Web

Connecting Discovery Islands Across Enterprises, SMBs, and Individuals

Agent discovery is diverging into multiple useful approaches, a natural consequence of innovation across different communities, use cases, and deployment environments. AI Catalog is an ideal starting point for public enterprise agent and AI-resource discovery, but it needs to be strengthened by allowing other discovery mechanisms to connect through it: DNS-AID, ANS, .well-known, A2A cards, MCP, gateways, and platform registries. NandaIndex is a Federated Resolution Architecture, a global switchboard that uses the AI Catalog format itself as the common index-record format, making it the preferred framework through which other discovery systems can be bridged.

Introduction

The agentic web is emerging rapidly. Enterprises are beginning to expose public AI agents, tools, skills, gateways, MCP servers, and catalogs. At the same time, SMBs and individuals will soon want discoverable agents without operating enterprise-grade infrastructure.

Innovation drives divergence. Different communities solving real problems, enterprises managing compliance, developers building open tools, governments standing up sovereign registries, telecom operators integrating agents into their networks, will naturally arrive at different discovery systems. This is not a coordination failure. It is what healthy ecosystems look like.

The answer is not to consolidate into one system. The answer is to build bridges.

AI Catalog is a strong starting point. When an enterprise publishes an AI Catalog at a well-known endpoint and controls the domain, direct resolution already works, and NandaIndex is not in the critical path. But AI Catalog does not cover every actor or every discovery system. Meanwhile, SMBs and individuals need paths to discoverability that require no enterprise infrastructure at all.

What is NANDA Index

NandaIndex is a federated index of AI Catalog-formatted resolution records. It introduces no new schema; each entry is an AI Catalog-compatible record that acts as a typed pointer, mapping an identity to a discovery entry point. Rather than acting as a centralized registry, NandaIndex is a distributed index of pointers: authoritative discovery remains with each system, and federation enables cross-boundary discovery.

ConceptImplementation
EntryAI Catalog-compatible index record
IdentityURN / URI identifier, plus a human-readable displayName
TargetA url, or inline routing data
Describedescription, tags, and publisher
Routing hintsMetadata fields (org.projectnanda.*)

A NandaIndex entry can point to any of the following targets:

  • AI Catalog record or .well-known URL
  • DNS-AID record
  • AGNTCY Agent Directory (ADS) entry
  • ANS record or DID
  • Enterprise gateway
  • A2A Agent Card or MCP descriptor
  • Hosted SMB or personal agent card
  • Telecom, IoT, EdgeAI, or sovereign national registry

Permissionless deployment

An identity does not require a unified infrastructure stack. The agent runtime can live on one provider (AWS, Azure, GCP, or another provider), the agent card can be hosted by a separate third-party card host (for example host39.org), and the brand domain can sit on yet another platform. No single party controls all three. No enterprise gateway or DNS certificate is required.

What is host39.org

host39.org is a third-party agent-card host. It lets SMBs and individuals publish agent-card information even when they do not own a DNS domain or cannot run catalog infrastructure on their own domain. The runtime can live with one provider and the brand domain with another; host39.org serves the agent card, and a NandaIndex entry points to it. The hosted-card example below resolves exactly this way.

Not in the critical path

When an enterprise publishes AI Catalog at a well-known endpoint and owns the domain, direct resolution already works. Any requester fetches the catalog directly, and a NandaIndex entry is optional. It remains useful for fallback, federation visibility, migration, and anti-squatting, but it is not required. NandaIndex adds value at the edges: actors that cannot publish a well-known catalog, discovery systems that are not AI Catalog-compatible, and federation across registries.

Anatomy of an entry

Every entry is an AI Catalog-compatible index record. It carries a stable identifier and a human-readable displayName, the mediaType of the target, the target itself (a url or inline data), a description and tags, the publisher, and metadata routing hints. This entry targets a hosted agent card by URL; the runtime, the card host, and the domain are three separate providers. 

{
  "identifier": "urn:ai:domain:moonbakery.com:agent:orders",
  "displayName": "Moon Bakery Orders Agent",
  "mediaType": "application/a2a-agent-card+json",
  "url": "https://agentcards.host39.org/moonbakery.com/orders.json",
  "description": "Ordering agent for Moon Bakery. Supports menu lookup, order placement, pickup scheduling, and order status.",
  "tags": ["smb", "bakery", "orders", "commerce", "a2a-agent-card"],
  "publisher": {
    "identifier": "urn:ai:domain:moonbakery.com",
    "displayName": "Moon Bakery",
    "identityType": "dns"
  },
  "metadata": {
    "org.projectnanda.resolutionRole": "smb-agent-card",
    "org.projectnanda.preferredDiscovery": "nandaindex",
    "org.projectnanda.agentCardHost": "host39.org",
    "org.projectnanda.runtime.provider": "AWS",
    "org.projectnanda.runtime.url": "https://moonbakery-orders.aws.example.com",
    "org.projectnanda.auth.metadata": "public",
    "org.projectnanda.auth.execution": "payment_or_session_token_required"
  }
}

Try Resolve

Pick an identity and resolve it through the switchboard. NandaIndex maps the identity to a discovery object and returns its routing hints; the requester then reads the discovery object, the agent card for endpoint and auth, and finally calls the runtime. NandaIndex routes the first hop and then steps aside.
