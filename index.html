A global switchboard for the Internet of AI Agents.

The web is poised to host billions of autonomous agents that negotiate, delegate and migrate in milliseconds. The NANDA Index resolves any agent identity to dynamic, cryptographically signed AgentFacts — a minimal, DNS-inspired layer for discovery, authentication and trust, without abandoning existing web infrastructure.

Why

DNS was built for sites, not agents

A TLS certificate proves you own a domain. It says nothing about an agent's code, its behaviour, or whether its capabilities are real. As workloads shift from minute-to-hour update cycles to millisecond orchestration, five pressures break the current stack.

01

Index writes & delay

DNS, BGP and WHOIS were built for millions of static records, not billions of updates per hour or sub-second worldwide sync.

02

Trust gap

Owning a domain is not the same as proving an agent's behaviour. The web needs signed capability proofs and instant revocation.

03

Privacy & split-horizon

Today's lookups expose who is asking for what. Enterprises need private, split-horizon resolution and tamper-evident audit logs.

04

Routing limits

Fixed A and AAAA records cannot keep up with agents that move every few seconds or need geo-based DDoS shuffling.

05

Governance & liability

Without a transparent, append-only log, no one can prove compliance or assign blame when agents misbehave across borders.

THE BET

One index, not a rebuild

Decouple identity from metadata, support multi-hop resolution, and provide a flexible substrate for discovery, routing and capability negotiation.

What

Lean record · verifiable AgentFacts

NANDA pairs a lean record holding only essential static metadata with a self-describing AgentFacts schema (signed JSON-LD) that lets agents update capabilities, endpoints and authentication without ever touching the index. In effect the index is a global switchboard: it connects a request to the right discovery object, then steps out of the way. The result is a lightweight, horizontally scalable foundation for secure, trust-aware collaboration, with five concrete guarantees.

1

A quilt-like index

NANDA-native agents and third-party agents from existing registries are both discoverable through one index.

2

Rapid global resolution

Newly spawned agents become resolvable worldwide in under a second of global propagation.

3

Sub-second revocation & key rotation

Credential revocation and key rotation propagate in under a second through a VC status list.

4

Schema-validated capability assertions

Capabilities are validated against a schema and signed as W3C Verifiable Credentials, not self-asserted prose.

5

Privacy-preserving discovery

Least-disclosure, unlinkable lookups work across organisational boundaries without leaking who asked.

The quilt. NANDA does not need to authorize or govern every agent on the web. Commercial, government and individual registries decide what is visible; the index merely holds a verifiable pointer to the right discovery object.

How it works

Three layers

The architecture cleanly separates static identity, verifiable metadata and dynamic routing. That separation cuts index write load by roughly 10,000x while keeping global resolution under a second.

LAYER 01 · ANCHOR

Lean Index

A decentralized mapping from agent IDs to a tiny AgentAddr (about 120 bytes) holding a DID, TTL, pointers and an Ed25519 signature. Cacheable, signed, tamper-resistant.

LAYER 02 · METADATA

AgentFacts

Self-describing JSON-LD documents, each a W3C Verifiable Credential, carrying endpoints, capabilities and auth. Agents update them independently, split across a primary and a privacy-preserving URL.

LAYER 03 · ROUTING

Dynamic Resolution

Latency, privacy and resilience-aware endpoint selection: static endpoints (TTL 1 to 6h), an adaptive resolver for geo load-balancing and DDoS mitigation, and rotating endpoints (TTL 5 to 15m).

Resolve

One identity, four hops

Every agent identity resolves through the same sequence — whatever path it registered by. The switchboard routes the first hop, then steps aside: hops 1–3 cache at the record TTL, and after hop 4 NANDA leaves the critical path entirely.

urn:ai:domain:acme.com:agent:time
1

NANDA Index

Who manages this identity?

in  locator: urn:ai:domain:acme.com:agent:time
out IndexRecord { registry_url, media_type, ttl }
GET /api/v1/resolve?locator=…
2

Agent Source

What is this agent’s entry?

in  IndexRecord.registry_url + identifier
out CatalogEntry { url, displayName, tags }
GET <registry_url>/agents/<id>
3

Agent Card

What can this agent do?

in  CatalogEntry.url
out AgentCard { url, auth, skills, … }
GET <catalog_entry.url>
4

Talk with Agent

Send a task directly.

in  AgentCard.url + { message }
out Task result from agent runtime
POST <agent_card.url>/run
